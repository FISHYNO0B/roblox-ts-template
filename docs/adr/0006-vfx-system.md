# Server-authoritative VFX as a Reflex slice + client-only camera shake

VFX replication is a top-level Reflex slice (`effects.list: Effect[]`) instead of a one-shot Networking event. The server dispatches `playEffect({ id, kind, cframe, startedAt, ttl })`; the broadcaster syncs the slice to every player; the client `VFXController` subscribes, clones the prebuilt template from `ReplicatedStorage/VFX/<folder>/<name>`, positions it, and `Debris:AddItem`s it after `ttl`. Camera shake is a separate concern — `vfx.shake({ intensity, duration })` is client-only state on the controller, runs in `RenderStepped`, never crosses the wire.

**Why a slice, not an event (and why this overrides [ADR 0001](0001-reflex-for-state-with-server-broadcast.md)'s "ephemeral signals stay as direct events"):** the parenthetical in ADR 0001 was a forward-looking hedge written before any concrete VFX work. With the spec in front of us, the rejoin-resilience case dominates: a player joining mid-fight should see the explosion that's already burning, and a slice gives that for free — the broadcaster hydrates new joiners with the current `effects.list` and the controller's "spawn on insert" subscription handles them like any other insert. With a one-shot event, late joiners get nothing. The "every effect is a slice action" cost is real but bounded — `expireEffects(now)` runs server-side every 0.5s on `Heartbeat` and trims the list, so the steady-state size is `(spawn rate) × (max ttl)` even under heavy combat.

**Why one controller for play and shake (D2):** same domain (visual feedback), no shared state across the boundary, and `shake` is a five-line `RenderStepped` loop. Splitting it into `CameraShakeController` would split docs and make `vfx.shakeFromExplosion(pos, radius)` reach into a sibling controller. Easier to fold in.

**Why a lazy per-key pool (D3):** repeated effects of the same key are the common case (hit, footstep, muzzle flash). Cloning + destroying every play churns instances and triggers full template prefetch. The pool stores inactive clones parented to a hidden folder; `play` pulls one (cloning if the pool is empty), positions it, and on TTL returns it instead of destroying. First play of a key still pays the clone cost; subsequent plays are O(1). Pool is unbounded — if a game spawns 1000 unique effects of the same key in a frame the pool grows accordingly, but that's a game-side problem, not a system one.

**Why server-generated GUIDs for effect ids (D4):** the slice is a list, so the controller needs a stable key to match insert ↔ remove. `HttpService:GenerateGUID(false)` is collision-free without coordination. An incrementing counter would work too but adds a server-side singleton that has to survive `resetState()` in tests.

**Why default ttl from the registry, callable override (D5):** registry holds the design-time default (`hit: 1.5s`); call sites occasionally need a one-off (a buff that should linger as long as the buff itself). `vfx.play("hit", cframe, { ttl: 5 })` covers both without forking the API.

**Why `ReplicatedStorage/VFX/<folder>/<name>` (D6):** matches the registry shape (`folder + name`) one-to-one. Per-game asset content is out of scope for the template — the template ships one stub model so the system is observably wired, and games sync their own from `assets/VFX/` via Rojo.

**Trade-offs:**
- **`effects.list` grows under spam.** Heartbeat cleanup runs every 0.5s; in the worst case (60 spawns/s × 3s ttl = 180 entries) the list is fine, but a runaway loop without ttl bound could leak. Mitigation lives at the call site, not the system.
- **CFrame in slice payload.** Reflex actions cross the broadcaster as Flamework Networking payloads — CFrame serializes natively, but every action carries a 12-component matrix even for effects that only use position. Acceptable; collapsing to `Vector3` would lose orientation for beam/attached effects.
- **Pool has no eviction.** A pool of 200 unique effect keys keeps 200 clones parented forever. For the template's expected usage (≤20 keys) this is fine; if a game grows past that, add an LRU cap.
- **Shake state is not save-game.** Pausing the game mid-shake and rejoining loses the shake. By design — shake is camera-frame feedback, not progression.
- **No `vfx.playAttached` in v1.** Spec mentions it; not in acceptance criteria. Adding it later means an `attachTo: Instance` field on the action, but Instances don't replicate cleanly through Reflex actions — would need a separate path (likely a Networking event for the attached subset). Deferred until a use case lands.
