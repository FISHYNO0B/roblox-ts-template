# Centralize all asset references in typed registries

Every `rbxassetid` reference in the project — icons, sounds, music, VFX, animations, UI images — lives in a typed registry under `src/shared/domain/assets/`. Components and services consume keys (`Icons.coin`, `Sounds.click`, `VFX.hit`) instead of raw ids. The `Icon` component takes `icon: IconKey`, never a raw string or number.

**Why:** Raw `rbxassetid://...` strings and bare digit ids were scattered across components and feature modules (currency frame, holder buttons, kit stories). Swapping an asset meant grepping across the codebase, and there was no way to attach metadata (sound group + volume, music loop flag, VFX folder/name + ttl) without inventing per-feature shapes. A single source of truth makes asset swaps a one-line edit and gives the upcoming Phase 1 systems (Sound, Music, VFX, Preload) a stable place to index into.

**Trade-off:** Each registry decides its own value shape — simple strings for `Icons`/`Images`/`Animations`; rich objects for `Sounds` (`{ id, group, volume }`), `Music` (`{ id, volume, looped }`), `VFX` (`{ folder, name, ttl }`). This is more cognitive load than one uniform shape, but the metadata each system needs is genuinely different, and a uniform shape would either be empty stubs or a dumping ground. Adding a new asset now also means picking the right registry — but the registries are narrow enough that the choice is obvious. The `Icon` API is a hard break: there is no compat shim for raw ids, so any new caller has to add an entry to `Icons.ts` first. That's the whole point.
