# Music as a stack-based controller, not a single-track service

`MusicController` owns a stack of `MusicKey`s, exposed via `push`, `pop`, `replace`, `stop`, `current`. The stack is capped at 4 entries. State transitions go through a pure reducer in [`shared/domain/music/stack.ts`](../../src/shared/domain/music/stack.ts) so the logic is unit-testable without a Roblox runtime. Sound instances are managed inside the controller — two `Sound`s coexist during a crossfade (one tweens to `0` and self-destroys on `Tween.Completed`, the other fades up to `entry.volume`). Volume routes through the existing `Music` `SoundGroup` bootstrapped by [`SoundController`](../../src/client/controllers/SoundController.ts) — the music slider is already wired by the [sound system](../systems/sound.md).

**Why a stack, not a single-current variable:** the typical caller flow is _menu → combat → boss → back to combat → back to menu_. A single-current model forces every caller to remember what was playing before, hand-rolling its own undo stack at the call site. A stack folds that into the system: `pop()` returns to whatever was on top before. The cap of 4 is high enough for nested contexts (menu / area / encounter / cutscene) but low enough that runaway pushes can't leak.

**Why a pure reducer:** the spec called for a logic-only test. Putting the state machine in a free function (`applyMusicStackOp`) means the test runs without `SoundService`, `TweenService`, or any Roblox dependency, and the controller's responsibility narrows to "play whatever the reducer says is on top." Same shape that Reflex already uses elsewhere in the project — pure transitions tested in isolation.

**Why no Reflex slice for current track:** D4. Music is transient client-side state. Persisting it would mean every rejoin resumes whatever was playing during the last session — fine for an audio app, weird for a Roblox game where the player expects the menu track on join. `DEFAULT_MUSIC` ("menu" by default, set to `undefined` to disable) is the only join-time decision the system makes.

**Why a network event for the Cmdr command, not client-side Cmdr:** the existing Cmdr setup is server-only ([`server/cmdr/startup.server.ts`](../../src/server/cmdr/commands/playMusic.ts)). Adding a client-side Cmdr registry would expand the surface area for one debug command. Instead, `playMusic <key>` runs on the server, calls `SoundService.playMusicFor(executor, key)`, which fires a `playMusic` event the controller picks up via `replace`. The event is server→client only and not part of the public music API — feature code uses `Dependency<MusicController>().push(key)` directly.

**Trade-offs:**
- **Crossfade overlaps both volumes.** During the fade window both old and new tracks contribute to `SoundGroup.Music.Volume × sound.Volume`. With linear tweens this peaks slightly above either track alone in the middle of the crossfade. Acceptable for typical 1s fades; if it ever becomes audible, swap the easing for `Sine` (in/out) so the curves intersect at half-volume.
- **`Sound` instances created per transition.** Each `push`/`replace` creates a fresh `Sound` instance and the previous one is destroyed on tween completion. No pooling. Music transitions are infrequent (seconds-to-minutes apart); the GC cost is negligible compared to a per-key cache that would also have to track loop state and current playback position.
- **Stack cap silently drops.** `push` past 4 entries shifts the oldest off the bottom. The dropped key is lost — `pop` won't return to it. Acceptable for misuse-prevention; if a game legitimately wants deeper nesting, raise `MUSIC_STACK_MAX`.
- **No `duck(factor)` API.** D3 — out of scope. If a game wants menu-open ducking, add `setDuck(factor: number)` that scales the in-flight sound's volume on top of `entry.volume`. The current API doesn't preclude it.
